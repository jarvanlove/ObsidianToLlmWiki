# ObsidianToWiki

面向 Obsidian 的 LLM 持续维护型知识系统。

它不是一个单纯的笔记模板仓库，也不是一个只会搜索的脚本集合。它的目标是把“原始资料、项目过程、稳定结论、可复用经验”持续整理成一个能长期维护、能跨会话恢复、能跨项目复用的本地 wiki。

## TL;DR

- 这是一个把个人知识和项目记忆统一起来的本地 wiki 系统
- 它支持项目接入、资料摄入、知识检索、问答回写、共享提升和持续治理
- 用户面向自然语言使用，系统内部负责脚本、索引、同步和维护
- 它适合长期使用，不适合把知识只停留在聊天窗口或散乱目录里

## 适合谁

这套系统尤其适合：

- 有长期知识积累需求的个人用户
- 同时维护多个项目、容易丢上下文的开发者
- 想把项目经验沉淀成共享方法的人
- 希望让 Codex / Claude Code / 其他 agent 持续使用同一套项目记忆的人

## 5 分钟上手

第一次使用，先做这 4 件事：

1. 下载 `ObsidianToWiki`
2. 准备一个私有库 `ObsidianToWiki-private`
3. 初始化私有库，让它具备入口、脚本、模板和最小目录结构
4. 完成一次真实使用：
   - 接入一个项目
   - 摄入一份资料

然后以后就主要用自然语言：

- `帮我把当前项目接入 wiki`
- `把这份资料收进当前项目`
- `基于当前项目 wiki 回答这个问题`
- `把这次结论记下来`

## 核心能力总览

```text
输入层
原始文件 / 临时资料 / 项目来源
    ->
记忆层
个人知识 / 项目知识 / 共享知识 / 输出沉淀
    ->
自动化层
项目接入 / 资料摄入 / 搜索召回 / 回写沉淀 / 治理体检 / 私有库同步
    ->
结果
任意窗口可恢复上下文 / 知识可持续积累 / 经验可跨项目复用
```

你可以把它理解成一个持续循环：

- 新资料进入系统
- agent 协助整理和链接
- 稳定结论写回 wiki
- 可复用经验提升到共享层
- 后续项目继续复用这些知识

## 产品定位

ObsidianToWiki 解决的是两个常见问题：

1. 个人知识长期积累
2. 软件项目长期记忆维护

它希望把下面这些分散内容，统一收进一个可持续演化的知识系统里：

- 原始文件
- 临时资料
- 项目上下文
- 架构说明
- 决策记录
- 任务推进状态
- 复盘与分析结论
- 可跨项目复用的方法、提示词、模式

一句话理解：

代码仓库负责交付物，wiki 负责记忆、解释、回写和复盘。

## 设计思想

这套系统的设计思想有 6 个核心点。

### 1. Markdown 页面是事实来源

最终知识不是存在数据库里，也不是只存在对话里，而是落在 Markdown 页面里，便于人读、人改、人长期维护。

### 2. 输入层和知识层分离

资料先进入输入层，再逐步沉淀到正式知识层，避免“所有东西都直接进知识库”导致结构失控。

### 3. 项目记忆和个人知识并存

它既服务个人知识积累，也服务项目级长期记忆，不把两者混成一个扁平笔记堆。

### 4. 用户只表达需求，不记技术命令

用户面向自然语言使用系统，脚本、规则、索引重建和日志维护属于系统内部能力。

### 5. 项目可以在任意窗口恢复上下文

项目只要接入一次，后续在任意编码窗口打开项目时，都可以通过项目根目录的上下文文件自动回到对应项目 wiki。

### 6. 知识要能提升和复用

项目中的好经验，不应该永远困在项目目录里，而应该能提升到共享层，服务后续项目。

## 功能模块

从产品视角看，这套系统主要有 8 个功能模块。

### 1. 项目接入

把一个现成项目正式接入 wiki，建立项目仓库和知识库之间的绑定关系。

接入后会生成：

- `wiki.context.json`
- `AGENTS.md`
- `CLAUDE.md`
- 对应的项目子 wiki

### 2. 资料摄入

把文档、PDF、会议纪要、方案、截图等资料接进系统，保存原始文件，并生成来源笔记。

### 3. 项目记忆维护

围绕项目维护：

- 概览
- 架构
- 决策
- 任务
- 来源
- 风险
- 时间线
- 运行记忆

### 4. 个人知识沉淀

把对个人长期有效的认知、方法论、工作流和经验沉淀到个人知识层。

### 5. 共享知识提升

把可跨项目复用的内容提升到共享层，例如：

- 架构模式
- 工作模式
- 提示词
- 工具使用经验

### 6. 搜索与问答回写

系统不仅支持找知识，还支持把高价值回答反过来写回 wiki，避免知识只停留在聊天窗口。

### 7. 治理与体检

系统会持续检查：

- 孤儿页
- 过期页
- 重复主题
- 未沉淀来源
- 结构问题

### 8. 私有库同步

公开脚手架更新后，可以把脚手架层内容同步到私有知识库，保证真实使用环境和当前系统保持一致。

## 架构设计

这套系统可以理解成三层架构。

### 1. 输入层

这一层负责接住原始输入：

- 原始文件
- 剪藏资料
- 临时中间产物
- 项目来源文件

对应目录：

- `01_inbox/`
- 项目内 `sources/`
- 项目内 `source-notes/`

### 2. 记忆层

这一层负责沉淀长期结构化知识：

- `10_personal/`
- `20_projects/`
- `30_shared/`
- `40_outputs/`

这是系统真正的知识主体。

### 3. 自动化层

这一层负责让系统持续运转：

- 页面创建
- 资料摄入
- 项目接入
- 搜索召回
- 问答回写
- 关系同步
- 来源同步
- 体检治理
- 私有库同步

对应目录：

- `00_system/scripts/`
- `00_system/templates/`
- `00_system/registry/`

- 模板
- 最小目录结构

### 第三步：完成第一次真实使用

第一次只做两件事：

1. 接入一个真实项目
2. 摄入一份真实资料

### 第四步：以后都按自然语言使用

## 项目怎么使用这套系统

这是系统最重要的使用场景之一。

### 项目只接入一次

每个项目只需要正式接入一次。接入后，项目仓库根目录会生成：

- `wiki.context.json`
- `AGENTS.md`
- `CLAUDE.md`

这三个文件就是“项目仓库 -> 项目 wiki”的桥。

### 任意窗口都能继续使用

以后无论从哪个窗口打开这个项目，只要窗口能访问项目根目录，agent 就可以先读取这些文件，再进入项目 wiki。

推荐读取顺序：

1. `wiki.context.json`
2. `AGENTS.md` 或 `CLAUDE.md`
3. 项目 `索引.md`
4. 项目 `project.memory.md`
5. 项目 `任务.md`

### 开发过程中怎么沉淀知识

推荐工作方式：

1. 做任务前先读项目记忆
2. 做任务时如有新资料，先进入项目来源层
3. 做任务后把稳定结论写回项目 wiki

回写重点通常是：

- `概览.md`
- `架构.md`
- `决策.md`
- `任务.md`
- `来源.md`

## 多模态支持

当前稳定支持：

- 文本
- Markdown
- 代码
- `docx`
- `pptx`
- `pdf`

图片、语音、视频建议分阶段支持：

1. 先完成入库和来源登记
2. 再补 OCR、转写、关键帧
3. 最后再自动沉淀到项目页或个人页

## Inbox 分类标准

当前建议按作用分类：

- `raw`：原始文件
- `clips`：尚未完全沉淀的来源笔记
- `scratch`：临时中间产物
