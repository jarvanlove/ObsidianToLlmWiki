# 使用手册

这份手册不追求讲全，只讲你日常会反复做的事情。

## 一句话理解

这套系统的工作方式是：

- 资料先进 `01_inbox` 或项目来源
- agent 帮你整理成项目页、共享页、输出页
- 以后靠搜索和索引再找回来

## 你只需要知道的目录

- `01_inbox/`
  新资料和临时材料
- `20_projects/`
  每个项目一套自己的知识区
- `30_shared/`
  跨项目共用的知识
- `40_outputs/`
  问答、分析、复盘结果
- `00_system/`
  系统内部脚手架

## 日常 4 个动作

## 1. 接入项目

适用场景：

- 你有一个现成代码仓库
- 希望 Codex / Claude Code 打开项目时能自动找到中心 wiki

你只需要说：

- `帮我把当前项目接入 wiki`

结果：

- 项目仓库会生成 `wiki.context.json`
- 项目仓库会生成 `AGENTS.md`、`CLAUDE.md`
- 私有 wiki 会生成项目主干页

## 2. 摄入资料

适用场景：

- 新需求文档
- 会议纪要
- 架构图说明
- 聊天记录整理稿

你只需要说：

- `把这份资料收进当前项目`
- `把这份资料收进个人知识库`

结果：

- 原文件会保存
- 来源笔记会创建
- 日志和索引会更新

当前系统对文件类型的支持可以这样理解：

- 文本 / 代码 / Markdown：直接收、直接提取
- `docx` / `pptx` / `pdf`：直接收、尽量提取正文
- 图片 / 语音 / 视频：先正式入库并登记为媒体来源，再调用你的多模态 LLM / API 来识别和理解

## 3. 搜索知识

适用场景：

- 我记得之前做过，但忘了在哪
- 我想知道某个项目依赖了什么
- 我想找权限、缓存、部署这类主题

你只需要说：

- `基于当前项目 wiki 回答这个问题`
- `帮我找一下之前关于权限设计的知识`

说明：

- 普通关键词搜索已经够大多数使用
- 如果问题和项目有关，系统会顺带带出项目关系和运行记忆

## 4. 写回结论

适用场景：

- 一次问答有长期价值
- 一次分析以后还会再用
- 一个项目结论不该只停留在聊天窗口

你只需要说：

- `把这次结论记下来`
- `把这次结论沉淀成个人知识`
- `把这个经验提升成共享知识`

说明：

- 如果你在项目窗口里说“把这次结论沉淀成个人知识”，系统应优先写入个人层，而不是默认回项目页
- 这类沉淀仍然应该保留和当前项目的关联

## 用自然语言发指令

你以后不需要优先记脚本名，可以先直接说需求。

例如：

- `帮我接入 wiki`
- `把这份资料收进当前项目`
- `把这份资料收进个人知识库`
- `把这张图片识别后沉淀到个人知识库`
- `把这张架构图整理后写进当前项目 wiki`
- `边开发边沉淀`
- `基于当前项目 wiki 回答`
- `把这次结论记下来`
- `把这次结论沉淀成个人知识`

更完整的话术清单见 [[标准自然语言话术清单]]。

系统里的实际处理原则是：

- 你说需求
- agent 识别这是接入、摄入、搜索、回写还是提升
- agent 同时判断最终写回目标是项目层、个人层、共享层还是输出层
- agent 调用内部能力并更新正确页面

如果你说得足够明确，系统就不应该再要求你去记脚本名、参数或目录细节。

这些属于 agent 该替你记住的实现细节。

## 项目窗口里的最小说法

如果你在任意项目窗口里，只想快速开始，可以直接说：

```text
请按当前项目的 wiki 规则工作。
先读取项目根目录的 wiki.context.json、AGENTS.md 或 CLAUDE.md。
完成任务后，把有长期价值的内容写回项目 wiki。
```

这就是项目场景下最小自然语言启动方式。

## 多模态资料怎么处理

当前最稳的是：

- 文本、Markdown、Office、PDF：直接摄入并提取
- 图片、语音、视频：现在已经能正式入库并登记为媒体来源，后续优先接入你的多模态 LLM / API 解析链路

目前的建议流程：

- 图片：先入库到 `01_inbox/raw/personal/images/`，再 OCR 和图片说明
- 语音：先入库到 `01_inbox/raw/personal/audio/`，再语音转写和摘要
- 视频：先入库到 `01_inbox/raw/personal/video/`，再转写音轨、抽关键帧、做时间轴摘要

也就是说，多模态资料先别要求“一步到位自动沉淀”，先保证“能入库、能登记、能继续加工”。

当前来源笔记会额外记录：

- `media_type`
- `parse_status`

而知识库体检会提示哪些媒体来源还处在待处理状态。

图片、语音、视频这类多模态内容，当前更推荐这样理解：

- 用户自己的多模态 LLM / API 负责识别和理解
- wiki 系统负责把原件、来源笔记、解析结果和沉淀页组织起来
- 不要求用户为这套 wiki 再额外安装本地 OCR / ASR / 视频处理工具

也就是说：

- “看懂图片/语音/视频”主要靠你的模型能力
- “把结果变成可维护知识”主要靠这套 wiki 系统

## 多模态只有一种推荐使用方式

### 会话内解析，默认这样用

这是你日常最应该用的方式。

适用场景：

- 你已经在 Codex / Claude Code 里
- 或者在 Cursor、Trae、QClaw、WorkBuddy、Hermes Agent、OpenClaw 等已集成模型能力的工具里
- 你把图片、语音、视频文件路径告诉当前 AI
- 让 AI 直接解析内容，并沉淀到项目 wiki 或个人 wiki

这条路的特点是：

- 不需要单独配置多模态 provider
- 不需要额外 API key 配置文件
- 直接复用当前会话模型的多模态能力

也就是说，如果你只是想说：

- `把这张图片识别后沉淀到个人知识库`
- `把这段语音整理后写进当前项目 wiki`

那么默认应该走这条路。

## Inbox 怎么判断放哪

只记这条就够了：

- 原件放 `01_inbox/raw/`
- 尚未完全沉淀的来源笔记放 `01_inbox/clips/`
- 临时中间文件放 `01_inbox/scratch/`

判断规则：

- 原始文件：`raw`
- 来源笔记：`clips`
- OCR / 转写 / 临时摘要：`scratch`
- 稳定知识：离开 inbox，进入正式知识层

## 项目目录怎么理解

一个项目目录里，你只先记住这些页面：

- `索引.md`
  项目入口
- `概览.md`
  这个项目是干什么的
- `架构.md`
  系统结构和关键模块
- `决策.md`
  为什么这么做
- `任务.md`
  当前在推进什么
- `来源.md`
  资料入口
- `project.memory.md`
  给 agent 快速读的运行态摘要

其他像 `关系.md`、`风险.md`、`时间线.md`，是在你开始稳定使用后再逐步补强的。

## 个人知识关系怎么写

个人知识层现在开始支持两个最小关系字段：

- `related_to`
- `builds_on`

最简单的理解：

- `related_to`
  这页和哪些个人知识页主题相关
- `builds_on`
  这页建立在哪些旧知识或旧方法之上

你不用一开始写很多。通常只要在真正重要的个人知识页里补 1 到 3 个关系就够了。

系统当前已经会做两件事：

- 重建 `10_personal/关系索引.md`
- 在检索结果中补充个人页的关系摘要

下一阶段开始，系统还会保守地做自动建议：

- 从你已经写出的个人页链接里回填关系
- 用共现标签和共同来源项目补充低风险关系

也就是说，后面你不一定每次都要手填全部关系字段。

当前这条链已经有第一版：

- `sync_personal_relations.py` 会同步个人关系字段
- `10_personal/关系索引.md` 会聚合这些关系
- 搜索个人知识时会显示关系摘要

## 如果你自己开一个新会话，应该怎么说

最简单的说法是：

```text
请按当前 wiki 规则工作。
先读 README-zh.md、快速开始.md、使用手册.md。
如果这是某个项目任务，再读取该项目仓库根目录的 wiki.context.json 和 AGENTS.md。
完成任务后，如果有长期价值，把结果写回 wiki。
```

更完整的模板见 [[会话启动页]]。

## 什么时候用私有库，什么时候用公开仓库

- 看脚手架、改规则、改模板、改系统行为：在当前公开仓库
- 放真实项目内容、个人知识、原始资料、分析沉淀：在 `ObsidianToWiki-private`

## 你暂时不用管什么

先不用深究这些：

- schema 注册表
- 自动候选发现
- 候选升级
- 版本状态报告
- 私有库同步 manifest

这些是系统治理能力，不是你第一次用起来的门槛。

## 你如果想知道脚本都在干嘛

不用一口气看完全部脚本，先按下面理解：

- `attach_project.py`
  负责“接项目”
- `ingest_source.py`
  负责“收资料”
- `handle_nl_request.py`
  负责“自然语言入口”
- `search_wiki.py`
  负责“查知识”，当前还会做页面类型权重、低价值结果降权、关系摘要补充和零结果记录
- `file_back_query.py`
  负责“写回结论”
- `lint_wiki.py`
  负责“体检”
- `sync_private_vault.py`
  负责“同步私有库”

其他脚本大多是这些主链路的增强或治理能力。

## 当前检索怎么抗退化

系统当前没有直接走重型检索基础设施，而是先靠这些方式抗规模增长：

- 分层检索：优先按项目层、个人层、共享层、输出层缩小范围
- 页面类型权重：更偏向概览、架构、决策、任务、概念、模式等稳定知识页
- 低价值结果降权：进一步压低反思候选、来源页、过期输出页
- 关系摘要补充：在项目、个人、共享结果旁补关系信息
- 失败记录：对零结果查询写日志，后续可用于学习候选和检索优化

当前阶段暂时不做：

- 分块检索
- 主题召回
- 向量检索

## 学习候选现在怎么理解

系统现在不只是会“发现学习候选”，还开始区分候选边界。

当前候选会带这些信息：

- 风险等级
- 升级方式
- 重复出现次数
- 新鲜度
- 所属领域

这意味着：

- 低风险候选可以逐步进入半自动处理
- 高风险候选仍然应该人工审批
- 低分且过期的候选可以被归档，而不是一直堆在候选池里

当前还补上了两层审核入口：

- `40_outputs/学习候选审批视图.md`
  集中看待审核、已批准和高风险候选
- `review_learning_candidates.py`
  对候选做批量批准、归档或重新打开
