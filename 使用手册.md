## 这套 Wiki 到底怎么用

把它理解成两层：

- `10_personal`：长期会反复复用的知识，比如工作流、架构模式、方法论、认知总结
- `20_projects`：每个具体项目的现场记忆，比如需求、架构、决策、经验、来源、周报、复盘

你平时不是“手工写 wiki”，而是：

1. 新项目开始时，先建项目 wiki
2. 看到资料时，先摄入成来源笔记
3. 让 Codex 或 Claude Code 把来源转成结构化页面更新
4. 做项目时，顺手把新决策、新经验、新风险写回项目页
5. 一段时间后，把可复用经验提升到 `30_shared`
6. 需要回忆时，用 `index + search_wiki`

更完整的可复制指令见 [[会话启动页]]。

## 数字前缀是什么意思

顶层目录前面的 `00`、`01`、`10`、`20`、`30`、`40`、`90` 有两个作用：

1. 控制排序  
   Obsidian 和文件系统默认按名称排序，加数字前缀后，核心分区会稳定排在固定位置。
2. 预留扩展空间  
   不写成 `1、2、3、4`，是为了以后插入新分区时不用整体重命名。

当前约定：

- `00_system`：规则、模板、脚本、维护工具
- `01_inbox`：进入知识库之前的原始材料和临时内容
- `10_personal`：长期有效的个人知识
- `20_projects`：项目级知识库
- `30_shared`：跨项目复用的知识资产
- `40_outputs`：分析、简报、问答沉淀
- `90_archive`：低频访问但保留价值的内容

## 为什么目录是英文，但依然不难用

目录和脚本名使用英文，是为了：

- 适合开源仓库
- 跨平台更稳定
- 便于协作和引用

但你的日常阅读界面仍然可以保持中文：

- 页面标题继续用中文
- 页面正文继续用中文
- Obsidian 链接显示名继续用中文

也就是说，英文主要存在于“文件系统层”，中文主要存在于“知识表达层”。

## 当前支持的资料格式

### 可直接提取正文

- `md`
- `txt`
- 常见纯文本代码和配置文件：`py/js/ts/json/yaml/yml/html/css/java/go/rs/sql`
- `docx`
- `pptx`
- `pdf`

### 当前只做入库登记，不稳定提取正文

- 老式 Office 格式，如 `doc`、`ppt`
- 图片类文件
- 压缩包、二进制产物、音视频文件

说明：

- 这些文件都可以被摄入到知识库，原文件会保存，来源笔记会创建，日志和索引会更新
- 只有上面“可直接提取正文”的格式，会自动把部分正文写入来源笔记
- `pdf` 已支持文本提取，但如果 PDF 是扫描件或图片型 PDF，结果仍可能较差

## 最小使用方式

### 第一步：建一个项目

Windows:

```powershell
.\00_system\scripts\new_project.ps1 -项目名 "demo-saas" -标签 "saas,ai"
```

macOS / Linux:

```bash
./00_system/scripts/new_project.sh --title "demo-saas" --tags "saas,ai"
```

如果你已经有一个现成项目仓库，希望把它快速接到中心 wiki，可直接执行：

```powershell
.\00_system\scripts\attach_project.ps1 -仓库根目录 "C:\path\to\repo" -项目名 "demo-saas" -Wiki根目录 "C:\path\to\private-vault"
```

这会在项目仓库中生成：

- `wiki.context.json`
- `AGENTS.md`
- `CLAUDE.md`

同时会在 wiki 里创建或确认该项目的主干页面，包括：

- `索引.md`
- `概览.md`
- `架构.md`
- `决策.md`
- `任务.md`
- `来源.md`
- `关系.md`
- `风险.md`
- `时间线.md`
- `project.memory.md`

如果不传 `-Wiki根目录`，脚本会优先查找当前公开仓库同级的 `ObsidianToWiki-private`，让项目默认接到私有知识库，而不是公开脚手架。

### 第二步：把资料丢进来

Windows:

```powershell
.\00_system\scripts\ingest_source.ps1 -源文件 "<资料路径>/prd.docx" -标题 "PRD v1" -项目 "demo-saas" -标签 "prd,需求"
```

macOS / Linux:

```bash
./00_system/scripts/ingest_source.sh --source "/path/to/prd.docx" --title "PRD v1" --project "demo-saas" --tags "prd,需求"
```

这会把原文件复制进项目目录，创建来源笔记，并自动更新索引和日志。

来源笔记现在会附带一组状态字段：

- `source_hash`
- `ingest_status`
- `derived_pages`
- `review_due`

这组字段是后续来源状态机、自学习和自动治理的基础。

当其他页面开始引用某个来源笔记后，可以运行：

```powershell
python 00_system/scripts/sync_source_notes.py --rebuild
```

它会自动回填：

- `derived_pages`
- `ingest_status`

`lint_wiki.py` 现在也会在体检前自动执行这一步。

如果你希望系统进一步为来源页推荐“该沉淀到哪里”，可以执行：

```powershell
python 00_system/scripts/recommend_source_promotions.py --rebuild
```

它会回填来源页的 `recommended_targets`，例如：

- `file-project:<project-slug>`
- `link-shared:<tag>`
- `promote-shared:<tag>`
- `promote-pattern:<tag>`

### 把问答直接沉淀回 Wiki

如果一次问答已经形成稳定结论，可以直接写回知识库：

```powershell
python 00_system/scripts/file_back_query.py --title "权限方案评估" --question "当前项目该用 RBAC 还是 ABAC？" --conclusion "先用 RBAC，保留 ABAC 扩展点。" --project "demo-saas" --tags "auth,analysis"
```

不指定 `--project` 时，会落到 `40_outputs/analyses/`；指定 `--project` 时，会落到对应项目的 `notes/`。

### 同步跨项目关系

如果你在项目 `关系.md` 中维护了依赖、复用、输出和相关项目，可以执行：

```powershell
python 00_system/scripts/sync_project_relations.py --rebuild
```

它会把 `关系.md`、`架构.md`、`决策.md`、`project.memory.md` 中的结构化链接同步回项目 `索引.md` frontmatter，并重建 [20_projects/关系索引](20_projects/关系索引.md)。

### 记录学习候选

如果你发现一个系统层面的改进点、重复问题或值得演化的工作流，可以先记录成候选反思：

```powershell
python 00_system/scripts/record_learning_candidate.py --title "查询失败模式-权限设计" --observation "多次查询权限方案时仍需手工补充上下文" --improvement "增加权限领域的共享模式页与别名词表"
```

不指定 `--project` 时，会写到 `40_outputs/reflections/`；指定 `--project` 时，会落到对应项目 `notes/`。

### 把学习候选升级成正式资产

如果某条候选已经确认有复用价值，可以直接提升：

```powershell
python 00_system/scripts/promote_learning_candidate.py --source "40_outputs/reflections/查询失败模式-权限设计.md" --target-type 模式
```

当前支持提升到：

- `模式`
- `提示词`
- `工具`
- `架构`
- `分析`

### 自动发现学习候选

如果希望系统从日志和体检报告中自动发现改进候选，可执行：

```powershell
python 00_system/scripts/discover_learning_candidates.py --rebuild
```

当前会从两类高信号来源生成候选：

- 最近体检报告中的结构性问题
- 日志中重复出现的更新模式
- 高频标签但共享层缺页的主题

如需压低候选噪音，可提高阈值：

```powershell
python 00_system/scripts/discover_learning_candidates.py --rebuild --min-score 5
```

### 同步公开脚手架到私有 vault

如果你已经在旁边维护 `ObsidianToWiki-private`，可先预览同步：

```powershell
python 00_system/scripts/sync_private_vault.py --dry-run
```

确认无误后再正式执行：

```powershell
python 00_system/scripts/sync_private_vault.py
```

如果你只想同步某一层，可加 `--only`：

```powershell
python 00_system/scripts/sync_private_vault.py --dry-run --only system --only prompts
```

这个脚本只同步脚手架层内容：

- 根入口文件
- `00_system/`
- `docs/`
- `30_shared/prompts/`

不会覆盖你的私有项目内容、个人知识或原始资料。

### 生成版本状态报告

如果你准备收口版本，可以生成当前公开仓库和私有 vault 的状态报告：

```powershell
python 00_system/scripts/version_status.py
```

默认输出到 `docs/plans/version-status.md`。

如果你准备正式进入提交阶段，可进一步生成按分组整理的收口报告：

```powershell
python 00_system/scripts/version_closure_report.py
```

默认输出到 `docs/plans/version-closure-report.md`。

### 第三步：让 Agent 处理资料

把 [[30_shared/prompts/资料摄入提示词|资料摄入提示词]] 发给 Codex 或 Claude Code，再补一句：

```text
请摄入 demo-saas 项目的 PRD v1，并更新概览、架构、决策、任务页。
```

### 第四步：边做项目边沉淀

```text
请在完成这个任务的同时，维护 demo-saas 的项目知识库。
```

## 常用操作

### 摄入资料

```powershell
.\00_system\scripts\ingest_source.ps1 -源文件 "<资料路径>/some-article.md" -标题 "某篇文章" -标签 "ai,workflow"
.\00_system\scripts\ingest_source.ps1 -源文件 "<资料路径>/prd.pdf" -标题 "PRD v1" -项目 "demo-saas" -标签 "prd,需求"
```

### 新建知识页

```powershell
.\00_system\scripts\new_page.ps1 -标题 "AI 编码工作流" -类型 概念 -标签 "ai,编码,工作流"
.\00_system\scripts\new_page.ps1 -标题 "权限方案评估" -类型 分析 -项目 "demo-saas" -标签 "auth,analysis"
```

### 项目过程页

```powershell
.\00_system\scripts\new_page.ps1 -标题 "权限系统 ADR-001" -类型 ADR -项目 "demo-saas" -标签 "auth,adr"
.\00_system\scripts\new_page.ps1 -标题 "第 16 周" -类型 项目周报 -项目 "demo-saas" -标签 "weekly,status"
.\00_system\scripts\new_page.ps1 -标题 "MVP 阶段复盘" -类型 项目复盘 -项目 "demo-saas" -标签 "retro,mvp"
```

### 搜索与查询

```powershell
.\00_system\scripts\search_wiki.ps1 -关键词 "权限 设计"
.\00_system\scripts\search_wiki.ps1 -关键词 "权限" -项目 "demo-saas"
.\00_system\scripts\search_wiki.ps1 -关键词 "鉴权" -类型 ADR
.\00_system\scripts\search_wiki.ps1 -关键词 "工作流" -标签 "ai"
```

如果希望项目查询时顺带带出依赖项目、复用资产和运行记忆，可执行：

```powershell
python 00_system/scripts/search_wiki.py "权限" --project demo-saas --show-relations
```

### 定期体检

```powershell
.\00_system\scripts\lint_wiki.ps1
```

体检结果会输出到 `40_outputs/analyses/知识库体检-日期.md`。

## 脚本和 Agent 到底是什么关系

不是让你以后手工靠脚本维护知识库，也不是脚本和 Codex / Claude Code 二选一。

更准确地说：

- `Codex / Claude Code` 是大脑，负责理解、判断、总结、更新页面
- 这些脚本是工具，负责做稳定、重复、低歧义的本地动作

为什么仍然需要脚本：

- agent 直接改文件当然可以，但同样的动作每次都自由发挥，稳定性会差
- 复制原始文件、创建来源笔记、追加日志、重建索引、按条件搜索这类动作，很适合固化成脚本
- 这样 Codex 和 Claude Code 都能调用同一套本地接口，行为更一致

所以以后最常见的模式不是你手工跑很多脚本，而是你对 agent 说：

```text
请把这个 PDF 摄入到 demo-saas 项目知识库，并更新相关页面。
```

然后 agent 自己去调用：

- `ingest_source.py`
- `search_wiki.py`
- `rebuild_indexes.py`
- `lint_wiki.py`
- `attach_project.py`
- `file_back_query.py`
- `record_learning_candidate.py`
- `promote_learning_candidate.py`

## 跨平台使用

Windows 使用 `.ps1`。  
macOS / Linux 使用 `.sh`。  
真正的核心逻辑在 `.py`，两套壳脚本只是不同系统上的轻量入口。

macOS / Linux 第一次使用前执行：

```bash
chmod +x 00_system/scripts/*.sh
```
